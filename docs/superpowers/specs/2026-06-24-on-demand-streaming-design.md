# On-Demand Streaming — Design

**Date:** 2026-06-24
**Status:** Approved

## Goal

Add a deployment mode where streams are pulled from the device **only while a viewer
is connected**, instead of the current always-on model where every enabled stream runs
continuously. This cuts device load and bandwidth to zero when nobody is watching.

## Background

Today `relay run` builds a `StreamManager` that logs in once and immediately starts a
RealPlay session + ffmpeg for **every** `enable: true` stream, keeping them all live until
shutdown (the "shared-login, always-on daemon" decided in the multi-camera design). On-demand
was explicitly deferred at that time; this spec implements it.

## Decisions

1. **Mechanism — MediaMTX `runOnDemand` (Option A).** MediaMTX natively supports lazy
   sources: when a viewer connects to a path that has no active publisher, MediaMTX *holds*
   the viewer connection, runs a configured command, waits up to `runOnDemandStartTimeout`
   for the stream to go live, and sends that command `SIGINT` once there are no readers for
   `runOnDemandCloseAfter` seconds. We point `runOnDemand` at a `relay serve <name>` process.
   This is event-driven (no polling) and lets MediaMTX own the entire start/stop/restart
   lifecycle.

2. **Per-active-stream login (not shared).** Each `runOnDemand`-spawned `relay serve` process
   does its own NetSDK login, streams one channel, and logs out when MediaMTX kills it.
   Rationale: dramatically simpler (no control channel, no dynamic daemon state, reuses the
   single-stream path), process-isolated, and MediaMTX-managed. The cost — N concurrently
   *viewed* cameras = N logins — is bounded precisely because on-demand only makes sense when
   you watch a subset at a time. Consistent with having dropped `login_limit` as YAGNI.

3. **Global toggle, not per-stream.** A single `ON_DEMAND` env var flips the whole deployment
   between always-on and on-demand. When set, **all** enabled streams become lazy and there is
   no long-running relay daemon — MediaMTX is the only engine. (Per-stream mixing was rejected:
   it would require running both the always-on daemon and MediaMTX `runOnDemand` simultaneously,
   a more complex two-engine entrypoint, for little real benefit.)

4. **Idle close-after = 10s** (MediaMTX default `runOnDemandCloseAfter`). Tears the device
   stream down quickly after the last viewer leaves.

5. **`uv run` everywhere.** The generated `runOnDemand` command and the container entrypoint
   both invoke Python through `uv run --frozen`, matching the project convention (AGENTS.md).
   `uv` is on `PATH` in Docker (`/usr/local/bin/uv`) and locally.

## Components

### New command: `relay serve <name>`

Serves a single on-demand stream, resolved by name from `streams.yml`:

- Load entries from `--config` (default `config/streams.yml`); find the entry whose `name`
  matches the positional `<name>` (which MediaMTX supplies as `$MTX_PATH`).
- If the name is absent or its entry is `enable: false`, log an error and exit non-zero
  (MediaMTX logs the failure; the viewer connection fails cleanly).
- Otherwise: log in, start RealPlay for `(channel, stream)`, publish to
  `cfg.publish_url(name)`, and block until `SIGINT`/`SIGTERM`, then stop the pipeline and
  clean up.

This is today's `cmd_stream` with the channel/stream looked up from config rather than passed
as flags. Extract a shared helper `_serve_one(cfg, channel, stream, name)` (login → RealPlay →
publish → block-until-signal → cleanup) used by **both** `cmd_stream` and `cmd_serve` (DRY).

A pure resolver supports it and is unit-tested:

```python
def find_enabled(entries: list[StreamEntry], name: str) -> StreamEntry:
    """Return the enabled entry with this name, or raise LookupError."""
```

### `ON_DEMAND` toggle on `Config`

Add `on_demand: bool = False` to `Config`, parsed from the `ON_DEMAND` env var using a truthy
check (`"1"`, `"true"`, `"yes"`, case-insensitive → True; unset/empty/other → False). Drives
two consumers:

#### `relay.mediamtx_config` (runtime config generation)

The MediaMTX config is generated by the entrypoint **before** MediaMTX starts, so the
on-demand paths must be emitted there. `build()` stays pure — it gains optional parameters and
the CLI does the env/file reading:

```python
def build(base, port, username, password,
          on_demand_names=None, run_on_demand_cmd=None,
          close_after="10s", start_timeout="10s") -> dict:
```

When `on_demand_names` is non-empty, `build()` adds:

```yaml
pathDefaults:
  runOnDemand: <run_on_demand_cmd>          # contains $MTX_PATH
  runOnDemandRestart: true
  runOnDemandCloseAfter: 10s
  runOnDemandStartTimeout: 10s
paths:
  <name1>: {}
  <name2>: {}
```

Only the listed (enabled) names get paths, so requests for unknown/disabled paths are rejected
by MediaMTX rather than spawning a doomed `serve`. **When `on_demand_names` is empty/None, the
output is byte-for-byte identical to today's** (no `paths`/`pathDefaults` block) — always-on
mode is unaffected.

The CLI (`python -m relay.mediamtx_config`) gains `--streams-config <path>` (default
`config/streams.yml`) and reads `ON_DEMAND` from the environment. When on-demand is on, it
loads the enabled stream names and builds `run_on_demand_cmd`:

```
uv run --frozen --directory <project_dir> python -m relay serve $MTX_PATH --config <abspath streams.yml>
```

- `<project_dir>` is derived from the relay package location
  (`dirname(dirname(relay.__file__))` → `/app` in Docker, repo root locally) so the command is
  cwd-independent.
- `<abspath streams.yml>` is the absolute form of `--streams-config`.

#### `cmd_run`

When `cfg.on_demand` is true, `cmd_run` does **not** start the `StreamManager`. Instead it:

- writes `output.m3u8` listing **all enabled** streams (they start when opened),
- logs the view URLs,
- idles until `SIGINT`/`SIGTERM`.

This keeps `relay run` as the container's foreground/signal-handling process, so the entrypoint
and `docker-compose.yml` need **no structural change** — flipping modes is purely
`ON_DEMAND=true` in `.env`. (In always-on mode `cmd_run` is unchanged: start the manager,
write the playlist from `manager.started`, idle until signal.)

### Entrypoint: `uv run` consistency

[deploy/entrypoint.sh](../../deploy/entrypoint.sh) already needs editing (the `mediamtx_config`
call gains `--streams-config`). While there, convert both Python invocations from
`/app/.venv/bin/python` to `uv run --frozen python …` for a single launch convention across
entrypoint, `runOnDemand`, and local dev. `--frozen` makes it a one-time, no-network env check
at container start. cwd is already `/app` (WORKDIR) so `uv` finds the project.

**Caveat to verify live:** the entrypoint backgrounds `relay` and traps `TERM/INT` to forward
to it for clean `docker stop`. After wrapping in `uv run`, `$PROXY_PID` is the `uv` process, so
clean shutdown relies on `uv` propagating `SIGTERM` to its child Python. Confirm
`docker compose stop` shuts down cleanly (no SIGKILL-at-timeout).

## Lifecycle

```
viewer opens rtsp://host:8554/cam3-main
  └─ MediaMTX (no publisher for cam3-main)
       └─ runs: uv run … python -m relay serve cam3-main --config …
             ├─ NetSDK login
             ├─ RealPlay ch3/main → ffmpeg -f dhav -c copy → publishes back to cam3-main
             ├─ MediaMTX unblocks the held viewer connection; frames flow
             └─ last viewer leaves → 10s (runOnDemandCloseAfter) → MediaMTX SIGINT
                   └─ relay serve: stop pipeline, logout, exit
```

## Testing

Pure logic is TDD'd; SDK/ffmpeg/end-to-end are validated live (channel 3 is known-good).

- **`mediamtx_config.build()`** — with `on_demand_names`: asserts `pathDefaults.runOnDemand`,
  `runOnDemandCloseAfter`, and a `paths` entry per name. Without it: asserts the output equals
  today's (regression guard), incl. that the auth block still works alongside.
- **`find_enabled(entries, name)`** — returns the matching enabled entry; raises `LookupError`
  for a missing name and for a present-but-disabled name.
- **`Config.on_demand`** — parsed from `ON_DEMAND` env: truthy values → True; unset/empty/
  arbitrary → False.
- **Live:** with `ON_DEMAND=true`, opening `rtsp://…/cam3-main` spins the stream up on connect
  (ffprobe sees frames within the start timeout); closing the viewer tears it down ~10s later
  (no `relay serve` process remains); a second open re-spawns it. Confirm `docker compose stop`
  is clean.

## Out of scope

- Per-stream on-demand selection (global toggle only).
- Shared-login on-demand (control-channel daemon) — rejected in favor of Option A.
- Tuning `runOnDemandStartTimeout` beyond the 10s default.
